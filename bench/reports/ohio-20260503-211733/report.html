Summary & recommendation

Across 23 chat models × 3 embedders × 20 questions (1380 judged cells, 0 errors), the bench reaches a clear verdict: granite3.3:2b delivers the best quality-per-byte and is the recommended default for ez-rag.

Caveat — factual accuracy: the rubric judge ranks this model #1, but the rule-based gold-truth check (must-contain factual phrases) gives it only 67%. deepseek-r1:32b hits 78% on the same check — a 11-point gap. Overall summary

  • Best score: granite3.3:2b at 10.95/12 (2.5B params, 3.4s/q).
  • Diminishing-returns knee: qwen3:1.7b at 9.90/12 is within 10% of the best using 1.5× fewer params.
  • Best cost-efficiency: deepseek-r1:1.5b at 4.30 score per second (score 8.27, 1.9s/q). Different from the absolute best — choose this when latency matters.
  • Reasoning models underperformed: the deepseek-r1 distillates averaged 9.13/12 vs 9.57/12 for non-reasoning models. The <think> blocks burn generation tokens that don't translate to better RAG answers on this corpus.
  • Embedder choice was a wash: all 3 embedders landed within 0.09 pts of each other. Pick by speed / VRAM cost, not by quality.

About this measurement (read me before quoting)

  • Single corpus. All scores reflect performance on a public-domain Ohio + Appalachian geology corpus assembled from sample_data/fetched/geology. Results may shift on technical documentation, code, fiction, or scientific corpora — model rankings often differ across domains.
  • 20 questions per (model, embedder). Mixed across factual lookups, comparisons, exploratory list queries, and multi-step synthesis. With this many questions, score differences smaller than ~0.4/12 are within noise.
  • Single judge model. qwen2.5:7b at temperature 0 scored every answer using a strict 0–12 rubric. The judge has its own biases — notably it tends to reward concise, citation-heavy answers over verbose ones.
  • Identical retrieval per (embedder × question). For each embedder we pre-computed retrieval once, then reused the same retrieved chunks across every chat model. So this measures generation quality only — not the chat model's own retrieval logic.
  • Ollama Q4_K_M quantization. All models tested at Q4_K_M. Higher precision (Q5, Q8, FP16) might change scores on the smallest models more than the largest ones.
  • One run per (model, embedder, question) cell. Temperature 0.2, no resampling. Models that errored once were retried once; two consecutive failures skip the rest of that model's questions for that embedder.
  • Power data is best-effort. GPU energy via nvidia-smi sampling at 500 ms intervals. CPU energy unmeasured on Windows / requires sudo on macOS. Numbers exclude the baseline idle draw, so absolute kJ should be read as a relative comparison, not literal wattage.

Confidence: Differences between adjacent ranks (e.g. 9.30 vs 9.37) should be treated as ties. Differences greater than ~0.5/12 are reliable.

Glossary · click to expand

Every defined term in this report has a hover-tooltip with the same text. Skim here for the full list.

addresses
Rubric dimension 1. Does the answer DIRECTLY answer the literal question, or does it pivot to a different topic?
auto-list mode
ez-rag auto-detects 'list X / give examples of X' queries and routes them through entity-rich HyDE + an extraction-only system prompt.
BM25
A keyword search algorithm (Okapi BM25). ez-rag uses it via SQLite FTS5 to find passages with literal term matches the dense vector search misses.
context window
The maximum number of tokens (~chars/3.5) the model can read at once. Includes the system prompt + retrieved chunks + chat history + answer reserve.
cost-efficiency
judge_total / seconds — how much quality you get per second of compute. Higher = better. Use this when picking a model for interactive chat where latency matters.
embedder
Model that converts text to vectors. Smaller and faster than the chat LLM. Each embedder uses a different vocabulary and dimension, so retrieval results vary across embedders even on the same corpus.
energy-efficiency
judge_total / kilojoule of GPU+CPU energy. Use this when picking a model for sustained throughput or battery-constrained deployments.
grounded
Rubric dimension 3. Is the answer drawn from the retrieved corpus content, or just from the model's training knowledge?
hybrid search
Running BM25 and dense search in parallel and fusing their rankings. The single biggest quality win over either method alone.
ingest
The one-time process of parsing source documents, chunking them, embedding the chunks, and storing both in a searchable index. Runs separately per embedder in this bench.
judge
An LLM scoring every answer with a strict 0-12 rubric. Same judge model for every cell so scores are comparable.
knee
Smallest model whose score is within 10% of the best — the diminishing-returns inflection point. Recommended pick for VRAM-constrained deployment.
on_topic
Rubric dimension 4. Does the answer stay on topic without drifting mid-paragraph to a different subject?
params
Trainable parameters (in billions). Roughly correlates with capability and VRAM cost. Q4_K_M quantization stores each parameter in ~4 bits, so VRAM ≈ params × 0.55 GB.
Q4_K_M
A specific 4-bit quantization scheme used by GGUF / Ollama models. Reduces VRAM by ~4× from full precision with minor quality loss.
RAG
Retrieval-Augmented Generation. The LLM is given relevant passages from a document corpus before answering, rather than relying on its training data alone.
reasoning model
An LLM trained to emit a hidden chain-of-thought (<think>...</think>) before its actual answer. Examples here: deepseek-r1:1.5b and 32b. The thinking blocks burn tokens but don't always improve RAG answers.
rerank
A small cross-encoder model that re-scores retrieval candidates with full attention to (query, passage) pairs. Adds ~50–200 ms per query for big quality wins.
rubric
Four 0-3 dimensions: addresses (does it answer the literal question?), specificity (concrete examples?), grounded (drawn from corpus, not training?), on_topic (no drift?). Sum is 0-12.
specificity
Rubric dimension 2. Does the answer contain specific named items / proper nouns / concrete examples, or only generic terminology?
VRAM
Video RAM on the GPU. The model's weights and per-token cache live here. Hard ceiling on model size at speed.
